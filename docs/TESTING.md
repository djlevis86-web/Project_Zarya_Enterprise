# Testing Guide

## Цель тестирования

Тесты должны защищать критическую ERP-логику проекта:

- права доступа;
- фильтрацию данных по пользователю;
- OCR-проверку сумм;
- платёжный реестр;
- частичные оплаты;
- экспорт;
- аудит действий.

## Базовый запуск

```bash
python manage.py test
```

## Проверка перед merge

```bash
python manage.py check
python manage.py makemigrations --check --dry-run
python manage.py migrate --plan
python manage.py test
```

## Приоритет тестов

### 1. Права доступа

Проверить:

- обычный пользователь видит только свои счета;
- обычный пользователь не может открыть чужой счёт через URL;
- staff-пользователь видит нужные административные данные;
- закрытые действия недоступны без авторизации.

### 2. Реестр оплаты

Проверить:

- счёт можно добавить в черновик реестра;
- счёт можно удалить из черновика;
- счёт с неподтверждённой OCR-суммой нельзя добавить в реестр;
- экспорт не включает лишние счета.

### 3. Частичные оплаты

Проверить:

- полная оплата даёт статус оплачено;
- частичная оплата даёт статус частично оплачено;
- отсутствие оплат даёт статус не оплачено;
- переплата не ломает расчёт остатка.

### 4. OCR

Проверить:

- OCR-задача создаётся;
- повторная постановка не создаёт лишних дублей;
- ошибка OCR сохраняется в статусе задачи;
- неподтверждённая сумма блокирует оплату.

### 5. Контрагенты

Проверить:

- контрагент определяется по данным OCR;
- неизвестный контрагент попадает в список unmatched;
- дубли не создаются без необходимости.

## Правило добавления новой бизнес-логики

Если добавляется новая бизнес-логика, вместе с ней должен появиться тест хотя бы на основной успешный сценарий и один граничный случай.
