# UI Redesign System Plan — Project Zarya Enterprise

## 1. Цель

Цель этого документа — зафиксировать системный подход к UI-редизайну проекта, чтобы больше не наращивать стили заплатками.

Мы не переписываем интерфейс хаотично. Сначала определяем компоненты, правила и этапы, затем меняем UI маленькими безопасными шагами.

---

## 2. Текущее стабильное состояние

Стабильной базой считаются ветки:

- main — продакшен-ветка;
- develop — интеграционная ветка после отката спорного UI-эксперимента;
- UI audit reports — сохранены в docs/ и scripts/.

Спорный этап page headers + empty states был отменён в develop через revert.

---

## 3. Почему нельзя продолжать CSS-заплатками

Проблемы текущего подхода:

1. Общие классы вроде page-header могут неожиданно менять внешний вид многих страниц.
2. Дополнительные CSS-файлы начинают переопределять старые стили.
3. Кнопки, бейджи, таблицы и карточки становятся визуально разными.
4. Каждая новая правка исправляет один экран, но может ломать соседний.
5. Потом приходится снова чистить стили.

Вывод: сначала нужна дизайн-система компонентов, потом внедрение.

---

## 4. Базовые UI-компоненты

Нужно привести к единому виду следующие элементы:

### 4.1. Page header

Единый верхний блок страницы:

- заголовок;
- описание;
- действия справа;
- опциональные метки/статусы.

Важно: нельзя менять все существующие header-классы одной общей CSS-правкой без проверки всех страниц.

### 4.2. Action bar

Панель действий должна отделять:

- статус;
- основные действия;
- вторичные действия;
- недоступные действия.

Статус не должен выглядеть как кнопка.

### 4.3. Buttons

Нужно определить единые варианты:

- primary;
- secondary;
- danger;
- ghost;
- small;
- disabled.

Кнопки в таблицах должны иметь единый размер и не прыгать по ширине.

### 4.4. Badges

Нужно унифицировать статусы:

- Новый;
- На проверке;
- Подтверждён;
- Оплачен;
- Частично оплачен;
- Не оплачен;
- Черновик;
- Выгружен;
- Ошибка.

Бейджи не должны быть похожи на кнопки действия.

### 4.5. Tables

Таблицы должны иметь единый паттерн:

- table container;
- table toolbar;
- header;
- rows;
- actions;
- empty state;
- pagination.

Отдельно нужно решить проблему широких таблиц и длинного текста.

### 4.6. Cards

Карточки должны иметь единый стиль:

- card;
- card-header;
- card-body;
- card-footer;
- summary-card;
- metric-card.

### 4.7. Forms and filters

Фильтры должны иметь единый layout:

- label;
- input/select;
- actions;
- reset;
- responsive behavior.

### 4.8. Payment mini-card

Блок оплаты встречается в нескольких таблицах.

Нужно сделать единый компонент:

- оплачено;
- остаток;
- статус оплаты.

### 4.9. OCR warning

Сообщения OCR должны быть читаемыми:

- сумма совпадает;
- сумма отличается;
- сумма требует проверки;
- оплата заблокирована.

Текст не должен вылезать из ячеек.

### 4.10. Toast/messages

Системные сообщения лучше показывать в одном месте:

- нижний левый угол;
- поверх интерфейса;
- единый стиль success/warning/error/info.

- page-header;

- registry-actions;
- card;
- table;
- status-badge;
- btn;
- payment-mini;
- ocr-badge.

Эти классы уже используются на многих страницах.

## 6. Предлагаемый порядок редизайна

### Этап 1. UI inventory

Собрать карту экранов и компонентов:

- dashboard;
- invoice list;
- invoice detail;
- upload;
- upload batches;
- OCR queue;
- payment schedule;
- payment registry;
- payment registry history;
- payment registry detail;
- counterparties;
- audit;
- system pages;
- users admin.

### Этап 2. Buttons and badges

Сначала унифицировать самые маленькие компоненты.

Почему:

- низкий риск;
- быстро видно результат;
- кнопки и статусы используются везде.

### Этап 3. Tables

После кнопок и бейджей приводим таблицы.

Цель:

- одинаковые отступы;
- одинаковые действия;
- нормальные переносы;
- нормальный responsive.

### Этап 4. Action bars

После таблиц приводим панели действий.

Особенно:

- реестр оплаты;
- список счетов;
- график платежей.

### Этап 5. Page headers

Только после понимания всех страниц приводим заголовки.

Не глобальной заплаткой, а контролируемо по шаблонам.

### Этап 6. Empty states

Добавляем пустые состояния после того, как таблицы и карточки уже стабилизированы.

### Этап 7. Toast/messages

Переносим системные сообщения в единый toast-контейнер.

---

## 7. Правила разработки UI

1. Один UI-этап — одна feature-ветка.
2. Не менять бизнес-логику вместе с UI.
3. Не менять права доступа вместе с UI.
4. Не добавлять новые CSS-заплатки без понимания, где уже используется класс.
5. Сначала искать все использования класса.
6. Перед правкой общего класса проверять все страницы, где он используется.
7. После каждого этапа запускать Django check.
8. После каждого этапа запускать UI smoke script.
9. Перед merge в main делать визуальную проверку локально.
10. На app.zarya35.ru отправлять только проверенный main.

---

## 8. Проверки перед merge

Перед merge в develop:

- python manage.py check;
- python .\scripts\ui_smoke_pages_v1.py;
- git diff --check;
- локальная визуальная проверка изменённых страниц.

Перед merge в main:

- production check;
- collectstatic;
- повторная smoke-проверка.

---

## 9. Что делаем первым после плана

Первый настоящий UI-этап после этого документа:

feature-ui-buttons-badges-v1

Цель:

- не трогать layout;
- не трогать page-header;
- не трогать большие таблицы;
- привести только кнопки и бейджи к единому виду;
- проверить результат на invoice_list, payment_registry, payment_schedule.

Это безопаснее, чем снова начинать с page-header.
